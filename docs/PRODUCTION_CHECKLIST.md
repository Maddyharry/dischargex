## ควรตรวจก่อนอัปโหลด

### 1. Environment (.env)
- [ ] **ไม่ commit `.env`** — ตรวจว่า `.env` ไม่อยู่ใน Git (มี `.env*` ใน .gitignore แล้ว)
- [ ] **Production:** ตั้ง `NEXTAUTH_URL=https://โดเมนจริง` (ไม่มี slash ท้าย)
- [ ] **AUTH_SECRET** — ใช้ค่าสุ่มใหม่ใน production (เช่น `openssl rand -base64 32`)
- [ ] **ADMIN_PASSWORD** — เปลี่ยนจากค่า default เป็นรหัสที่แข็งแรง
- [ ] **OPENAI_API_KEY** — ใช้ key ที่มี quota เพียงพอและจำกัดการใช้งานถ้าต้องการ

### 2. OAuth (ถ้าใช้ Google/Facebook)
- [ ] ใน Google / Facebook Console ใส่ **Authorized redirect URI** เป็น `https://โดเมนจริง/api/auth/callback/google` (และ facebook)
- [ ] ใส่ **Authorized JavaScript origins** เป็น `https://โดเมนจริง`

### 3. ฐานข้อมูล
- [ ] **Production DB** — ตอนนี้ใช้ SQLite (`file:./dev.db`) เหมาะสำหรับ dev/ทดลอง ถ้าขายจริงควรพิจารณาใช้ PostgreSQL หรือ MySQL บน hosting และเปลี่ยน `DATABASE_URL`
- [ ] รัน migration / `prisma db push` หลังเปลี่ยน DB

### 4. การชำระเงิน
- [ ] **QR PromptPay** — ใส่รูป QR ของคุณในโฟลเดอร์ `public/` (เช่น `promptpay-qr.png`) แล้วตั้งใน .env: `NEXT_PUBLIC_PROMPTPAY_QR_URL=/promptpay-qr.png` หน้าแพ็กเกจจะแสดง QR ให้ลูกค้าสแกนจ่าย แล้วยังคงแนบสลิปในฟอร์มเหมือนเดิม
- [ ] ตอนนี้ระบบรับคำขอผ่านฟอร์ม + อัปโหลดสลิป แล้วแอดมินอนุมัติเอง — เหมาะสำหรับช่วงเริ่มต้น
- [ ] ถ้าต้องการรับชำระอัตโนมัติ: พิจารณาต่อกับ Omise, Stripe, หรือ PromptPay API ตามนโยบายการรับเงิน

### 5. ข้อความและกฎหมาย (ถ้าต้องการครบ)
- [ ] หน้า **แนวทางใช้งาน** (`/guidelines`) มีแล้ว — ตรวจว่าข้อความเหมาะกับบริการจริง
- [ ] ถ้าขายเป็นธุรกิจ: พิจารณาเพิ่ม **ข้อกำหนดและเงื่อนไขการให้บริการ** และ **นโยบายความเป็นส่วนตัว** เป็นหน้าแยก หรือลิงก์ไป PDF

### 6. การติดต่อและสนับสนุน
- [ ] แชทช่วยเหลือ (ปุ่มมุมล่างขวา) ใช้ได้แล้ว — ตรวจว่าทีมมีวิธีรับแจ้งและตอบลูกค้า
- [ ] ถ้ามีอีเมล/เบอร์ติดต่อธุรกิจ ใส่ในหน้า About หรือ Footer ได้

### 7. Monitoring (ถ้าต้องการ)
- [ ] เปิด error tracking (เช่น Vercel Analytics, Sentry) เพื่อดู error หลัง deploy
- [ ] ตรวจ log ของ hosting ว่ามี 4xx/5xx เยอะผิดปกติหรือไม่
